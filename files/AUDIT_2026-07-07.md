## ČÁST A — OPRAVENO v tomto běhu (ověřeno typecheck + testy + build)

### Bezpečnost / write-path (`apps/api/src/powersync.ts`, `index.ts`)
1. **[VYSOKÁ] Projektová role se ve write-path nevynucovala.** `commenter` mohl PATCHovat cizí úkoly,
   **smazat celý projekt** (cascade) i **převzít vlastnictví** (`owner_id`). → Přidán registr
   `minRole`/`managerCols`; role se čte z `project_members.role`; DELETE projektu a změna
   `owner_id`/`visibility`/`status`/`archived_at` jen `manager`.
2. **[VYSOKÁ] Injekce push notifikace komukoli.** `reminders.user_id` byl klientem řízený a worker
   posílal push bez kontroly členství → útočník doručil libovolný text libovolnému uživateli. →
   `ownerCol: "user_id"` (server VŽDY dosadí session id) + přesun do per-user sync bucketu.
3. **[STŘEDNÍ] Zaseklá upload fronta / ztráta offline dat.** Klasifikátor SQLSTATE `^(22|23)\d{3}$`
   míjel `22P02` (neplatné UUID/enum) → vracelo 500 → PowerSync retry donekonečna, fronta stojí. →
   opraveno na `^(22|23)[0-9A-Za-z]{3}$` + autorizační dotazy zabaleny do try/catch (i chyba
   membership dotazu → 400).
4. **[STŘEDNÍ] Cross-bucket spoofing.** Denormalizovaný `project_id` (řídí sync bucket) se
   nevaliditoval proti projektu úkolu → řádek šel podstrčit do cizího bucketu. → server dopočítá
   `project_id` z `task_id` a klientskou hodnotu ignoruje.
5. **[STŘEDNÍ] Padělání audit-logu.** `task_activity.user_id`/hodnoty libovolné. → `appendOnly`
   (PATCH/DELETE zakázán) + `ownerCol` (autor = session).
6. **[STŘEDNÍ] Eskalace práv.** `manager` mohl udělit roli `admin` (nad svou úroveň). → cap: nelze
   přidělit roli vyšší, než má volající (role-change i invite).
7. **[STŘEDNÍ] Únik osobních připomínek.** `reminders` se syncovaly celé po projektu → každý viděl
   nastavení ostatních. → per-user bucket `WHERE user_id = bucket.user_id`.
8. **[NÍZKÁ] Barvy/komentáře.** `task_user_colors.user_id` padělatelný → `ownerCol`. Komentáře mohl
   editovat/mazat kdokoli → `authorEditOnly` (jen autor nebo manager).

> ⚠️ Po změně `sync-config.yaml` je nutný restart `docker restart watson-powersync` (jinak jedou
> stará sync pravidla).

### Ztráta dat / offline (`connector.ts`, `QuickAdd.tsx`)
9. **[KRITICKÁ] Offline fronta zahazovala zápisy při přechodných chybách.** `FATAL` bral i 401/408/429
   a při jedné vadné op `tx.complete()` potvrdil i NEODESLANÉ op. → přepsáno: trvalé chyby (400/403/
   409/422) přeskočí JEN svou op a pokračují; přechodné (401/408/429/5xx/síť) → retry celé transakce.
10. **[VYSOKÁ] QuickAdd neposílal `project_id` u přiřazení** → řádky se nikdy nesyncly (bucket je per
    projekt), kolegům se přiřazení nezobrazilo a po resyncu zmizelo i autorovi. → doplněn `project_id`
    + odvození `assignment_mode` (`shared_all` u ≥2, jinak `single`).
11. **[VYSOKÁ] QuickAdd na „Dnes" vkládal do abecedně prvního projektu** místo Schránky. → sdílený
    `pickInboxId(projects, activeWs)` v `lib/inbox.ts`.

### Doménová logika (`tasks.ts`)
12. **[KRITICKÁ] R2 `shared_all` byl čistě kosmetický** — jediný člen dokončil úkol všem a naopak
    „všichni odškrtli" úkol nikdy nedokončilo. → `toggleTask(task, actorId)` větví: u `shared_all`
    (≥2 přiřazení) přepíná JEN účast aktéra a `tasks.completed_at` je odvozené až jsou hotoví všichni.
    Zapojeno v TaskItem, TaskDetailPanel, kbNav (Space). *(Omezení: `shared_all` + opakování se
    zatím nechová jako řada — viz backlog.)*
13. **[VYSOKÁ→UI] Nástěnka míchala statusy všech projektů** (N×3 sloupce, drop zapsal cizí
    `status_id`). → sloupce se slučují podle názvu a drop se mapuje na status VLASTNÍHO projektu úkolu.

### Datum/čas (UTC → lokální) — „úkoly mizí po půlnoci"
14. **[VYSOKÁ]** `todayISO()` počítal dnešek v UTC na 8 místech (Today, Postupy, Reporty, Cíle,
    WatsonPanel, Nadcházející, Projekty, TaskDetailPanel) → mezi půlnocí a ~02:00 CEST úkol s dnešním
    termínem zmizel z Dnes. → sjednoceno na lokální čas (shodně s `lib/tasks.ts`).
    *(Engine `occurrences.ts` je záměrně čistě UTC-interní a otestovaný — neměněn.)*

### UX / drobné
15. **[VYSOKÁ]** Escape nezavíral „Přidat úkol" (guard hledal neexistující `data-add-layer`). → atribut doplněn.

### Výkon (`AppSchema.ts` klient, Drizzle migrace `0015`)
16. **[VYSOKÁ] Chybějící klientské SQLite indexy** — `tasks` full-scan na každou změnu. → přidány
    indexy `by_parent/by_due/by_completed/by_status` (tasks) a `by_user` (assignments).
17. **[STŘEDNÍ] Chybějící Postgres indexy** — migrace `0015_perf_security_indexes`: reminders
    (task/user + parciální `WHERE sent_at IS NULL` pro worker), comments/task_activity/assignments/
    overrides/colors `project_id`, `project_members.user_id`, `projects.workspace_id`.

## ČÁST B — PRIORITIZOVANÝ BACKLOG (větší, do tohoto běhu se nevešlo)

### P0 — škálovatelnost (bez toho appka nezvládne 30 lidí × desetitisíce úkolů)
- **Retence sync pravidel.** `task_activity` NEsyncovat (detail on-demand přes API); staré dokončené
  úkoly filtrovat (`completed_at > now() - 90 dní`); archivační mechanismus. **Určuje strop všeho.**
- **Scopované dotazy místo `SELECT *`.** Today/Ukoly/Nadcházející/Sidebar/rowMeta tahají celou tabulku
  a filtrují v JS. Přejít na `WHERE`/konkrétní sloupce/`LIMIT` (indexy z bodu 16 jsou připravené).
- **Virtualizace seznamů** (`@tanstack/react-virtual`) — dnes se renderují všechny řádky najednou.
- **`RowMetaProvider`** = 7 celotabulkových watched dotazů invaliduje celou app → přestavět na SQL
  agregace scopované na viditelné úkoly.

### P1 — správnost (souběhy, transakce, čas)
- **Server-authored advance postupů** (dnes klientský LWW → dva aktivní kroky / done úkol s active
  krokem při offline souběhu). Viz RECONCILIACE §23.
- **Lokální transakce** (`writeTransaction`) kolem vícekrokových operací (toggle+advance+reflow,
  create chainu, delete přes 6 tabulek) — pád uprostřed = nekonzistence.
- **`due_date`/`deadline` jako `date`/wall-clock**, ne `timestamptz` (round-trip posouvá den/čas dle
  TZ serveru). Přidat integrační test round-tripu.
- **R4 reset per-osoba** u opakování: `task_occurrence_overrides` nemá `user_id` (dokončení výskytu je
  globální); `shared_all` + opakování se nechová jako řada.
- **Undo sync-aware** — dnes „absolutní přepis", může přepsat novější změnu kolegy; nevrací chain advance.
- **Batch write endpoint** — 1 HTTP + 5–8 dotazů na každou op; hromadné akce (Přeplánovat) = N×.

### P2 — mobil, i18n, přístupnost
- **Mobil: půlka appky nedostupná** — Nastavení/Schránka/Cíle/Reporty/Postupy nejsou v `MobileTabBar`
  ani v (skrytém) sidebaru → potřeba tab „Více"; kompaktní header (dnes přetéká na 375px).
- **i18n úniky** — `lib/goals.ts` celý česky (propisuje se do EN), hardcoded aria-labely, formáty „do
  …dní", desetinná čárka natvrdo; **chybějící plurály** (`taskCount`, `daysUnit` → „1 úkolů", „2 dní").
- **Přístupnost** — modaly bez focus-trap/návratu fokusu; řádky úkolů klikací `div` bez role/tabindex;
  z-index vrstvení (detail se otevře pod panely Postupů/Reportů); sjednotit Esc kaskádu.
- **Drag&drop bez touch alternativy** (Board/Kalendář) — na mobilu nefunkční.

### P3 — bundle a provoz
- **Code-split rout** (`lazy()`) + ořez WASM precache (4 varianty `wa-sqlite`, `mc-*` šifrování se
  nepoužívá) → precache 8,8 MB → ~4 MB.
- Produkční tvrzení: rate-limit `/api/*`, odstranit dev tokeny (`service.yaml`), `requireEmailVerification`,
  reálné odeslání magic-linku/pozvánek (blokováno Blokem mail I).

## ČÁST C — VERDIKT

**Bezpečnost:** základní R5 architektura (bucket parameter queries, whitelist sloupců, parametrizované
SQL) je zdravá; slabinou byla **granularita autorizace** (role, vlastnictví polí) — opraveno.
**Škálovatelnost:** dnes OK do ~2 000 otevřených úkolů; **30 lidí × 50k úkolů současný stav neudrží** —
úzké hrdlo je klient (full-scany, žádná virtualizace) a objem syncovaných dat (`task_activity`).
Řešení = ČÁST B / P0. API + Postgres samotné 30 uživatelů zvládnou.
**Doménová správnost:** occurrence engine a quick-add jsou solidní; systémové slabiny byly
v dokončování úkolu (4 implementace) a v absenci transakcí/serverové reconciliace — hlavní (R2, R9,
Board) opraveno, zbytek v P1.
